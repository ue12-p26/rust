# Memory of a process

During execution of a program, memory is used to store code, constants and
data as well as manage function calls. A process' memory is divided into
several segments (see {numref}`fig-process-memory`):

- The *Text* segment stores *compiled code*.
- The *.rodata* segment stores *constants* (i.e.: read-only data).
- The *.data* segment stores *initialized static variables*.
- The *.bss* segment stores *uninitialized static variables*.
- The *Stack* stores *function calls* and *local variables*.
- The *Heap* stores *dynamic memory allocation*.

## Text (Code)

In the *Text* segment is stored the whole *machine code* generated by the
compiler.
This memory zone is set as *read-only* by the *Operating System*: trying to
write in this zone would trigger a *segmentation fault*.

## .rodata

The *.rodata* segment is where are stored constants.
As for the *Text* segment, this memory zone is set as *read-only* by the
*Operating System*: trying to write in this zone would trigger a
*segmentation fault*.

Declared constants are usually stored inside the *.rodata* segment, though
they may be *inlined* by the compiler for optimization:

```{code-cell} rust
const PI: f32 = 3.14159;
PI
```

Literal strings are stored inside the *.rodata* segment:

```{code-cell} rust
static GREETING: &str = "Hello, world!";
GREETING
```

## .data

The *.data* segment stores *initialized static variables*: static variables
that have initialization values at compile time.

Example with an integer:

```{code-cell} rust
static ZIPCODE: u32 = 92340;
ZIPCODE
```

:::{note} Immutable static or constant?
Since the static variable is immutable, the compiler may store it inside the
*.rodata* segment instead.
:::

Example with a structure initialized using a function:

```{code-cell} rust
#[derive(Debug)]
#[allow(dead_code)]
struct Point {
  x: i32,
  y: i32,
}

const fn new_point(x: i32, y: i32) -> Point {
  Point {x, y}
}

static ORIGIN: Point = new_point(0, 0);
ORIGIN
```

:::{note} Constant function
The function needs to be declared *const* in order to be used for static
initialization.
:::

:::{note} Attributes `derive` and `allow`
[Attributes](#chp-attr) are metadata attached to an object for the
compiler. The `allow` attribute is a compiler directive in order to
silence some warning. The `derive` is an *attribute-like* macro that
adds a feature, here the `Debug` trait, to a type using a default
implementation.
:::

## .bss

The *.bss* segment (Block Started by Symbol) is where are stored
*uninitialized static variables*: static variables that have **no** known
initialization values at compile time. However, the compiler, knowing the
sizes of those objects, allocates the needed size inside the generated
program's file. At runtime, the *Operating System* provisions the necessary
memory zone, and initialize it to *zeros*. Later on, during program's
execution, these data will eventually receive values.

In Rust, zero initialized variables are stored in the *.bss* segment:

```{code-cell} rust
static mut N: usize = 0;
static mut BUFFER: [u8; 16] = [0; 16];

#[allow(static_mut_refs)]
fn fill_and_print() {
  unsafe {
    // Initialize
    N = 10;
    for i in 0..N {
      BUFFER[i] = 5;
    }

    // Print
    println!("N={N}");
    println!("BUFFER={BUFFER:?}");
  }
}

fill_and_print();
```

## Stack

The stack is a region of memory that stores local variables created by each
function (including the main function). It operates in a last-in, first-out
(LIFO) manner, meaning that the most recently added function call is the
first one to be removed when it completes.

:::{warning} Stack size
There exists a *stack* for each *thread* of execution in a process.
Each process has one or more *threads*, their number may vary during
execution.
The *default* stack size of a *thread* is set by the operating system.
However a process may request a different stack size (higher or lower), and
the operating system may accept or not the request.
Usually, the default stack size is arround 1 or 2 MiB.
:::

Each time a function is called, a new **stack frame** is created on top of
the stack to hold (see {numref}`fig-memory-stack`):

- The return address (where to go back in the code after the function
  finishes).
- The received arguments.
- The local variables.

When the function returns, its stack frame is popped off the stack, and
execution is resumed at the return address.

In the following example, functions `foo_a()`, `foo_b()` and `foo_c()` are
pushed on the *stack*, as well as their variables/parameters `n`, `m` and
`o`:

```{code-cell} rust
fn foo_c(o: i32) -> i32 {
  o * 3 + 8
}

fn foo_b(m: i32) -> i32 {
  foo_c(m / 2)
}

fn foo_a(n: i32) -> i32 {
  foo_b(n + 5) * 2
}

foo_a(10)
```

### Stack overflow

If a program uses too much stack space, through too many nested function
calls and/or heavy usage of local variables, it can lead to a stack overflow
error, causing the program to crash, as show in
{numref}`fig-stack-overflow`.

:::{figure} media/memory_stack_overflow.svg
:name: fig-stack-overflow
:alt: Stack overflow
:width: 100%

Stack overflow
:::

In the following example we push a huge array on the *stack* that would
generate an *overflow* at runtime (not executed here to avoid crashing the
kernel):

```rust
let array = [123; 1000_000_000];
array.len()
```

### Call stack

The call stack is a conceptual representation of the active stack frames at
any given point during program execution. It shows the sequence of function
calls that led to the current point in the program, with the most recent
call at the top (see {numref}`fig-call-stack`).

## Heap

The heap (see {numref}`fig-memory-heap`) is a region of memory used for
dynamic memory allocation, where objects are stored and managed during the
program's execution. Unlike the stack, which has a strict LIFO structure,
the heap allows for flexible memory allocation and deallocation. Objects in
the heap can be created and destroyed in any order, and their lifetimes are
not directly tied to function calls.

:::{figure} media/memory_heap.svg
:name: fig-memory-heap
:alt: Memory heap
:width: 50%

Memory heap
:::

:::{warning} Heap size
Inside a process the *heap* memory is shared among all its *threads*.
Each time a thread needs more heap memory, it sends a request to the
operating system, which may accept or reject the request. The limit of the
heap memory is dependent on the machine hardware (RAM) and the operating
system. The *total amount* of available heap memory is shared among all
processes. If too much processes are running at the same time, using too
much memory, the operating system may decide, depending on its
configuration, to terminate some of them in order to avoid an overall
*slowdown* of the system or a *crash*.
:::

The memory heap grows according to the needs of the program. Each time more
memory is needed, the program sends a request to the operating system. The
theoretical maximum size of the heap is determined by the operating system,
architedture (32-bit vs 64-bit) and configuration, but is always larger
than the available physical memory, thanks to virtual memory techniques.
This is however aperformance bottleneck, as accessing data in RAM is much
faster than accessing data in virtual memory (on disk). Therefore, it is
important to manage memory usage efficiently to avoid excessive swapping
between RAM and disk.

The following example makes usage of *heap* objects:

```{code-cell} rust
let mut books: Vec<String> = Vec::new();
books.push(String::from("Le Petit Prince"));
books.push(String::from("Le avventure di Pinocchio. Storia di un burattino"));
books
```
